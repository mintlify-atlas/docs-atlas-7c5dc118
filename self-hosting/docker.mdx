OpenPanel ships a Docker Compose stack that runs every required service â€” reverse proxy, database, cache, event store, API, dashboard, and worker â€” in a single command. This guide walks through a full deployment from a blank Linux server.

## Prerequisites

- A Linux server (Ubuntu 22.04 recommended) with at least 2 vCPUs and 4 GB RAM
- A domain name with an A record pointing to your server's public IP
- Ports 80 and 443 open in your firewall
- Docker and Docker Compose v2 installed (the setup script can install them for you)

<Tip>
The interactive `setup` script can install Docker and Node.js automatically if they are not already present.
</Tip>

## Services overview

The Compose stack runs the following containers:

| Service | Image | Role |
|---|---|---|
| `op-proxy` | `caddy:2-alpine` | TLS termination and reverse proxy |
| `op-db` | `postgres:14-alpine` | Primary relational database |
| `op-kv` | `redis:7.2.5-alpine` | Queue and cache |
| `op-ch` | `clickhouse/clickhouse-server:25.10.2.65` | Analytics event store |
| `op-api` | `lindesvard/openpanel-api:2` | REST API, runs DB migrations on start |
| `op-dashboard` | `lindesvard/openpanel-dashboard:2` | Next.js frontend |
| `op-worker` | `lindesvard/openpanel-worker:2` | Background job processor |

All services use `restart: always` so they come back up after a server reboot. Log rotation is configured on every container (max 10 MB per file, 3 files).

## Setup

<Steps>

<Step title="Clone the self-hosting repository">

Download the self-hosting files from the OpenPanel repository:

```bash
git clone https://github.com/Openpanel-dev/openpanel.git
cd openpanel/self-hosting
```

The directory contains:
- `docker-compose.template.yml` â€” Compose template (the setup script copies this to `docker-compose.yml`)
- `.env.template` â€” Environment variable template
- `caddy/Caddyfile.template` â€” Caddy reverse-proxy template
- `clickhouse/` â€” ClickHouse configuration and database init script
- `setup`, `start`, `stop`, `restart`, `update` â€” Helper scripts

</Step>

<Step title="Run the interactive setup script">

The `setup` script installs dependencies, asks a few questions, and writes both `docker-compose.yml` and `.env` for you.

```bash
bash setup
```

The script will prompt you for:

1. **Domain name** â€” the full URL including protocol, e.g. `https://analytics.example.com`. Both `DASHBOARD_URL` and `API_URL` are derived from this value (`<domain>` and `<domain>/api` respectively).
2. **Which dependencies to run** â€” by default ClickHouse, Redis, and Postgres are all included in the Compose stack. You can deselect any to use an external managed service instead.
3. **Reverse proxy** â€” choose *Install Caddy with SSL* to have the script generate a `Caddyfile` with automatic HTTPS, or *Bring my own* to skip adding `op-proxy` to the stack.
4. **Worker replicas** â€” number of `op-worker` containers (the `OP_WORKER_REPLICAS` variable). Defaults to half your CPU count, minimum 1.
5. **Resend API key** (optional) â€” for outbound transactional email.
6. **Basic auth password** â€” protects the worker dashboard at `worker.<your-domain>`.

After the script completes, `docker-compose.yml` and `.env` are ready in the `self-hosting/` directory.

<Note>
If you prefer to configure everything manually, copy the template files and edit them directly:

```bash
cp docker-compose.template.yml docker-compose.yml
cp .env.template .env
```

Then fill in all `$VARIABLE` placeholders in `.env`. See the [Configuration](/self-hosting/configuration) page for every available variable.
</Note>

</Step>

<Step title="Review the generated Caddyfile">

If you chose the Caddy option, the setup script generates `caddy/Caddyfile` from the template:

```caddy
<your-domain> {
    encode gzip

    handle_path /api* {
        reverse_proxy op-api:3000
    }

    reverse_proxy /* op-dashboard:3000
}

worker.<your-domain> {
    encode gzip

    basic_auth {
        admin <bcrypt-hashed-password>
    }

    reverse_proxy op-worker:3000
}
```

All requests to `/api/*` are forwarded to `op-api:3000`. Everything else is forwarded to `op-dashboard:3000`. Caddy provisions and renews TLS certificates automatically via Let's Encrypt.

The worker dashboard is exposed at `worker.<your-domain>` and protected with HTTP basic auth using the password you chose during setup.

</Step>

<Step title="Start the stack">

```bash
bash start
```

This runs `docker compose up -d`. Docker pulls all images and starts the services in dependency order:

1. `op-db`, `op-kv`, and `op-ch` start first
2. `op-api` waits for all three to pass their health checks, then runs database migrations with `CI=true pnpm -r run migrate:deploy` before starting
3. `op-dashboard` and `op-worker` start after `op-api` is healthy
4. `op-proxy` starts last, after both `op-dashboard` and `op-api` are healthy

To watch the startup logs:

```bash
bash logs
```

</Step>

<Step title="Create your first admin account">

Once the stack is running, open your domain in a browser. Because `ALLOW_REGISTRATION` defaults to `false`, you need to enable it temporarily to create the first account.

Edit `.env` and set:

```env
ALLOW_REGISTRATION="true"
```

Then restart the stack:

```bash
bash restart
```

Register your admin account at `https://<your-domain>/register`, then **immediately** set `ALLOW_REGISTRATION` back to `false` and restart again:

```bash
bash restart
```

<Warning>
Leaving `ALLOW_REGISTRATION="true"` in production allows anyone to create an account on your instance. Disable it as soon as your first admin account is created. Use `ALLOW_INVITATION="true"` to invite additional users by email instead.
</Warning>

</Step>

</Steps>

## Managing the stack

The `self-hosting/` directory includes helper scripts for common operations:

| Script | Command | Effect |
|---|---|---|
| `start` | `bash start` | `docker compose up -d` |
| `stop` | `bash stop` | `docker compose down` |
| `restart` | `bash restart` | `docker compose restart` |
| `update` | `bash update` | Pull new images and recreate containers (see [Updating](/self-hosting/updating)) |
| `logs` | `bash logs` | Stream logs from all containers |

## Volume storage

All persistent data is stored in named Docker volumes:

| Volume | Contents |
|---|---|
| `op-db-data` | PostgreSQL data directory |
| `op-kv-data` | Redis AOF/RDB files |
| `op-ch-data` | ClickHouse database files |
| `op-ch-logs` | ClickHouse server logs |
| `op-proxy-data` | Caddy TLS certificates and ACME state |
| `op-proxy-config` | Caddy configuration data |

<Warning>
Do not delete these volumes unless you intend to wipe your data. Always back up `op-db-data` and `op-ch-data` before running major updates. See [Updating OpenPanel](/self-hosting/updating) for backup guidance.
</Warning>

## ClickHouse configuration

The `clickhouse/` directory contains two XML config files mounted read-only into the container:

- **`clickhouse-config.xml`** â€” Sets log level to `warning`, disables verbose internal query/metric logs, listens on `0.0.0.0`, and sets inter-server hostname to `op-ch`.
- **`clickhouse-user-config.xml`** â€” Disables per-query and per-thread logging for the default profile.
- **`init-db.sh`** â€” Runs on first container start to create the `openpanel` database:

```bash
clickhouse client -n <<-EOSQL
    CREATE DATABASE IF NOT EXISTS openpanel;
EOSQL
```

The `nofile` ulimit is set to 262144 (both soft and hard) as required by ClickHouse.

## Worker replicas

The `op-worker` service uses Docker Compose's `deploy.replicas` setting to control horizontal scaling:

```yaml
deploy:
  mode: replicated
  replicas: $OP_WORKER_REPLICAS
```

Set `OP_WORKER_REPLICAS` in your `.env` file. For most deployments, 1â€“2 replicas is sufficient. Restart the stack after changing this value.

<Note>
`deploy.replicas` is honoured by `docker compose up` in Compose v2. You do not need Docker Swarm mode.
</Note>
